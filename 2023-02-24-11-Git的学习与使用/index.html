第一个branch_name是指远程的分支名。第二个是本地的分支名。不指定可能会造成某种冲突。

pull、clone、fetch的区别
git clone是直接把远程仓库复制到本地，不需要git init。这也是我作为一个完全不懂git的小白最常用的操作。
git fetch是拉取远程分支，只拉取不合并，需要手动merge到本地分支
git pull = git fetch + git merge，直接将本地分支更新到远程版本。
使用pull和fetch可能会出现冲突，原因是远程版本和本地版本对同一内容有不同修改，解决方法是拉取到一个新的branch，手动解决后再merge。

推送到远程仓库

推送到别人的仓库
他人的仓库是无法直接push的（毕竟不能往别人的仓库乱扔东西）。
解决方法通常分为两种：

  • 1.请求别人将自己添加为collaborators。
  • 2.发起Pull Request\Merge Request,大家俗称的PR\pr（不是Adobe Premiere）

PR:申请对方拉取自己的代码。当想要向别人的仓库发送代码时，就需要发起PR。
一般你要需要写申请的理由，你做的内容等等。

issue(问题讨论)
提出项目的Bug，记录milestone。甚至可以作为Blog或者forum。

Gitignore——我不需要提交xxx

实际开发时，我们并不需要提交所有的文件。（那就是文件地狱了）
例如当我们用Visual Studio写了一个cpp程序，项目文件夹会非常大，但我们其实只需要提交其中的.hpp文件和.cpp文件即可。这时如果我们直接使用git add。将会提交很多不必要的文件。很恐怖的好吗

Gitignore的作用是设置文件的忽略规则，让Git根据该规则有选择地忽略文件的更改。

使用gitignore的方式便是在目录中创建名为.gitignore的文本文件，在该文本文件里编写忽略规则。

最常见的规则：
1.直接写上文件名。PS:重名文件需要给出目录
2.忽略文件夹
忽略所有名为bin的文件夹内的文件
bin/
忽略当前目录下名为bin的文件夹内的文件
./bin/
即也是和html一样的路径写法。

因为需要忽略的文件是可以找到共性的，一定有文件不需要上传。所以会有Gitignore模板。
Git语法和Gitignore模板都可以在github找到。

还有很多操作（通配符、反向操作）。想要学习的去RTFM吧！

git commit的行业规范（遵守的规矩）

git commit最大的价值就是标注信息，那如何才能最高效地标识清楚修改的内容就是其最大意义。

目前，社区有多种commit message的写法规范。本文介绍Angular规范是目前使用最广的写法，比较合理系统化，并且有配套的工具。前端框架Augular.js采用的就是该规范。

目的在于：

  • 可读性好、清晰，不必深入看代码就可以了解当前commit的作用。
  • 为code reviewing做准备
  • 方便跟踪工程历史
  • 让其他的开发者在运行git blame的时候非常感动
  • 提高项目整体质量，提高个人工程素质

按照标准，规定commit的文字部分格式：
格式要求包含三个部分：Header、Body和Footer；其中header是必须的，另外两者可以省略。

Header

该部分只有一行包含三个字段，type、scope、subject。
